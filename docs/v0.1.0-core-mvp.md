# NakedCanvas — v0.1.0 "Core MVP" Spec (P0)

> Status: **Draft / In Planning**
> Owner: PO + Engineering
> Milestone: `v0.1.0`
> Theme: Technical proof — _"Can pure, dependency-free code sustain 60fps interaction?"_

---

## 1. Objective

Prove that a from-scratch particle engine built on the raw HTML5 Canvas 2D API — with **zero external libraries** — can:

1. Sample a text string into thousands of particles.
2. Run a mouse-driven repulsion + elastic-return physics simulation.
3. Hold **60fps at 5,000 particles**.
4. Ship in a **≤ 2KB** minified + gzipped bundle.

This release is a **capability proof**, not a feature product. Public API surface is intentionally minimal and may change before `v1.0.0`.

## 2. Scope

### 2.1 In Scope (P0)

| #   | Capability                                                                            | Notes                                                      |
| --- | ------------------------------------------------------------------------------------- | ---------------------------------------------------------- |
| S1  | **Core interface** initialized by injecting an `HTMLCanvasElement`                    | `new NakedCanvas(canvas, options?)`                        |
| S2  | **Single-text pixel sampling** via an **offscreen temporary buffer** (`getImageData`) | Text is rasterized off-screen, never on the visible canvas |
| S3  | **Single-loop physics**: mouse repulsion + elastic return + friction                  | One flat pass per frame over the particle array            |
| S4  | Pointer input (mouse + touch) with enter/leave clearing                               |                                                            |
| S5  | Device-pixel-ratio (DPR) correct rendering **and** sampling                           | Must not break on Retina (dpr ≥ 2)                         |

### 2.2 Out of Scope (deferred to v0.2.0+)

- Multiple / concurrent text layers or image sources.
- Per-particle color variation, gradients, or theming.
- Lifecycle controls (`pause()`, `destroy()`, teardown of listeners).
- Configurable particle shape/size, connection lines, or trails.
- Framework bindings (React/Vue wrappers).
- SSR / worker-thread rendering.

## 3. Architecture Decision (P0 Critical)

### Decision: **Split the sampling buffer from the render canvas.**

Two separate 2D contexts, each optimized for its one job:

```
┌─────────────────────────────┐        ┌──────────────────────────────┐
│  Offscreen Sampling Buffer   │        │   Visible Render Canvas      │
│  ───────────────────────     │        │   ─────────────────────      │
│  getContext("2d",            │  read  │   getContext("2d")           │
│    { willReadFrequently:true})──pixels─▶  (GPU-accelerated, default) │
│  • draws text once           │        │  • fillRect() x N per frame  │
│  • getImageData() once        │        │  • never read back           │
└─────────────────────────────┘        └──────────────────────────────┘
```

**Why this is mandatory for P0:**

1. **Performance.** `willReadFrequently: true` forces the context into **software rendering** to make `getImageData` cheap. If that flag sits on the _render_ canvas, every one of the ~5,000 `fillRect` calls per frame loses GPU acceleration — directly sabotaging the 60fps target. The flag belongs **only** on the buffer we actually read from.
2. **Correctness (S2).** Rasterizing text onto the visible canvas causes a one-frame flash and pollutes the render surface. An offscreen buffer keeps the visible canvas clean.
3. **Correctness (S5).** Sampling and rendering coordinate spaces stop fighting: the buffer is sized in **device pixels** and read in device pixels; the render canvas handles DPR scaling independently. This removes the "only top-left 1/dpr region gets sampled" Retina bug.

> This single refactor satisfies S2, unblocks the 60fps target, and fixes the Retina sampling bug simultaneously.

## 4. Public API (P0)

```ts
interface NakedOptions {
  particleGap: number; // pixel sampling stride (lower = denser)
  mouseRadius: number; // interaction radius (px)
  pushForce: number; // repulsion strength
  returnEase: number; // elastic return factor (0.01–0.2)
  friction: number; // velocity damping (0.8–0.98)
  defaultColor: string; // uniform particle color (P0: single color only)
}

class NakedCanvas {
  constructor(canvas: HTMLCanvasElement, options?: Partial<NakedOptions>);
  renderText(text: string): void; // sample text + start the RAF loop
}
```

No other public methods ship in P0.

## 5. Acceptance Criteria

A build passes `v0.1.0` **only if all** of the following hold:

| ID  | Criterion                                                                           | Measurement                                        |
| --- | ----------------------------------------------------------------------------------- | -------------------------------------------------- |
| AC1 | Sustains **≥ 60fps with 5,000 particles** on a mid-tier laptop                      | DevTools Performance / rAF frame timing            |
| AC2 | Minified + gzipped bundle **≤ 2KB**                                                 | `vite build` output + `gzip -c dist/*.js \| wc -c` |
| AC3 | Text samples correctly on **dpr = 1 and dpr = 2** (no clipping)                     | Visual check on Retina + non-Retina                |
| AC4 | **No text flash** on the visible canvas during `renderText()`                       | Visual / frame capture                             |
| AC5 | Mouse repulsion and elastic return behave smoothly; particles settle back to target | Manual interaction                                 |
| AC6 | Touch input drives the same interaction; leave/end clears the pointer               | Manual on device / emulator                        |
| AC7 | `package.json` `dependencies` is `{}`; no runtime imports                           | Code + `package.json` review                       |
| AC8 | No `let`, no `any`, no `arc()`; RAF loop allocates nothing via `new`                | Static grep + review                               |

## 6. Performance Budget & Rules (from CLAUDE.md)

- **Zero-GC RAF loop:** no `new` object/array/instance creation inside the 60fps loop; pre-allocate or mutate in place.
- **Data-oriented:** particles are raw `ParticleData` structs in a flat array; no per-particle class instances or methods.
- **Native rendering:** use `fillRect()`, never `arc()`, for particles.
- **Math:** compare squared distances (`distanceSq < radiusSq`); avoid `Math.sqrt` except for the one normalization inside the interaction branch.
- **Uniform fillStyle:** with a single P0 color, set `ctx.fillStyle` **once outside** the particle loop (not per particle).
- **Build:** Vite lib mode, `target: "esnext"` — no polyfills / TS helpers in the bundle.

## 7. Risks & Mitigations

| Risk                                            | Impact | Mitigation                                             |
| ----------------------------------------------- | ------ | ------------------------------------------------------ |
| `willReadFrequently` on render canvas kills fps | High   | Offscreen buffer split (§3)                            |
| Retina sampling clips text                      | High   | Buffer sized in device pixels (§3)                     |
| Per-particle `fillStyle` cost at 5k             | Medium | Hoist fillStyle for uniform color (§6)                 |
| 2KB budget breached by build helpers            | Medium | ESNext target, no polyfills; measure in CI             |
| `getImageData` blocked by tainted canvas        | Low    | Only self-drawn text is read; no external images in P0 |
